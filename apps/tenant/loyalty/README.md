# Loyalty service-API (для ordering-BFF)

Server-to-server API лояльности: внешний ordering-BFF (приложение заказа
LevOne / Шавуха) берёт баланс / начисляет / списывает / возвращает баллы по
`vk_id`, не дублируя логику лояльности. Реализация ТЗ
`levone-newmobile/docs/LOYALUP_API_REQUEST.md`.

## Аутентификация
`Authorization: Bearer <LOYALTY_SERVICE_API_KEY>` (env, не в гите). Сравнение
постоянное по времени; без ключа в окружении — fail closed. Класс —
`apps.shared.users.auth.ServiceKeyAuthentication` (подключён ТОЛЬКО к loyalty-вью,
в глобальную DRF-цепочку не добавлен → гостевые JWT сюда не попадают).

Тенант определяется по хосту (как у всего API): `levone.levelupapp.ru` → схема
`levone`. `branch_id` — Branch внутри этой схемы.

## Решения владельца (зафиксированы)
- **1 балл = 1 ₽.**
- **Баланс — сетевой**: агрегат по всем точкам тенанта (бренда). `branch_id` —
  только «где оформлен заказ» (куда писать транзакцию / аналитика).
- **Частичная оплата — до 100%** (своего лимита нет; «не больше суммы заказа»
  валидирует чекаут BFF, здесь — только проверка баланса).
- **Начисление считает LoyalUP**: `floor(order_total * LOYALTY_ACCRUAL_PERCENT/100)`,
  по умолчанию `10%`. BFF может прислать явный `points` — тогда берётся он.
  ⚙️ Процент кэшбэка — единственный незакрытый бизнес-параметр; задаётся
  env `LOYALTY_ACCRUAL_PERCENT` (число). Менять без передеплоя кода.

## Идемпотентность
Каждый мутирующий вызов принимает `idempotency_key` (BFF генерит из order_id,
напр. `accrue:BFF-555`). Повтор с тем же ключом → прежний ответ, повторно НЕ
применяется. Журнал — `LoyaltyIdempotencyKey`; ключ пишется в той же транзакции,
что и монеты, поэтому гонка-дубль откатывает свои движения и отдаёт ответ
первого. Если `idempotency_key` не прислан — строится из `<op>:<order_id>`.

## Эндпоинты (tenant scope, `/api/v1/`)

| Метод | Путь | Назначение |
|---|---|---|
| GET  | `/loyalty/balance?vk_id=&branch_id=` | сетевой баланс |
| POST | `/loyalty/accrue` | зафиксировать заказ + начислить |
| POST | `/loyalty/redeem` | списать баллы (частичная оплата) |
| POST | `/loyalty/refund` | откат заказа (реверс начисления + возврат списанных) |
| GET  | `/loyalty/spend?vk_id=&branch_id=&period_days=90` | траты ₽ за период (статусы) |

Тела/ответы — см. `LOYALUP_API_REQUEST.md`. Доп. поле `order_id` в ответах —
для трассировки, можно игнорировать.

### Коды ошибок (`{ "detail": "...", "code": "..." }`)
`guest_not_found` (404), `branch_not_found` (404), `invalid_amount` (400),
`insufficient_balance` (409, + `balance`), `order_not_found` (404, refund).

## Что под капотом
- Сами монеты пишутся в `branch.CoinTransaction` (`source='delivery'`) — баланс,
  админка и мобайл-API видят их без изменений.
- `LoyaltyOrder` хранит лоялти-состояние заказа (сумма ₽ для статусов, начислено/
  списано, статус active/refunded). `spend` суммирует `order_amount` активных
  заказов; возврат исключает заказ из трат.
- `refund` реверсит начисление напрямую (не через `create_transfer`), поэтому
  откат корректен, даже если баллы уже потрачены (баланс может временно уйти
  в минус — это ожидаемо для корректировки).

## Деплой
Новые таблицы в КАЖДОЙ схеме тенанта (модели — TENANT_APP):
```bash
# локально/staging: создать миграцию заново необязательно — 0001 уже в репо.
python manage.py migrate_schemas            # применит loyalty.0001 во все тенант-схемы
# выставить ключ окружения перед рестартом web:
#   LOYALTY_SERVICE_API_KEY=<random-64>, опц. LOYALTY_ACCRUAL_PERCENT=10
docker compose up -d --force-recreate web   # .env читается только при create
```
Перед прод-миграцией — `pg_dump` бэкап (регламент). Миграция аддитивная
(только CreateModel) — существующих данных не трогает.

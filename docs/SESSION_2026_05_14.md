## Архитектура CheckUp relay

```
┌─ LoyalUP mobile (хозяин ресторана пишет) ─┐
│                                            │
└──────────────────┬─────────────────────────┘
                   │ POST /api/v1/support/chat/messages/
                   ▼
┌─ LoyalUP backend (Django, port 7000, в docker) ──────────────┐
│  • SupportChatMessage(sender=user) created                    │
│  • _safe_relay_to_checkup() ──┐                              │
└───────────────────────────────┼──────────────────────────────┘
                                │ POST https://checkupapp.ru/api/v1/loyalup/inbound/
                                │ headers: X-LoyalUP-Relay-Secret + (Host через nginx)
                                ▼ nginx → 127.0.0.1:8000 (daphne)
┌─ CheckUp backend (Daphne, port 8000, на хосте) ──────────────┐
│  • find/create Chat в LoyalUP-Support-Hub client (id=13)     │
│  • name="LoyalUP — <tenant_name>", chat_type=group           │
│  • участники из группы loyalup_support (id=2)                │
│  • create Message от service-user loyalup_relay (id=80)      │
│  • Channels WS broadcast ──┐                                 │
└────────────────────────────┼─────────────────────────────────┘
                             │
                             ▼
              CheckUp UI (web/mobile) — менеджер видит чат, отвечает
                             │
              ──────── менеджер пишет в чате ──────────
                             │
┌─ post_save signal на Message ────────────────────────────────┐
│  if chat is LoyalUP-relay AND sender != bot:                 │
│    POST http://localhost:7000/api/v1/                        │
│         internal/support/inbound-reply/                      │
│    headers: X-LoyalUP-Relay-Secret + Host: levelupapp.ru     │
└──────────────────┬───────────────────────────────────────────┘
                   │
                   ▼
┌─ LoyalUP backend ────────────────────────────────────────────┐
│  • InboundReplyView (apps/shared/relay/views.py)             │
```

## Уроки на будущее

1. **Никогда не делай `docker compose up --force-recreate web`
   если в overlay есть unsaved работа.** Сначала `docker cp` всё
   на host + `git commit + push`.
2. **После любого изменения** на проде — сразу `git add → commit
   → push`. Без исключений (это в CLAUDE.md как «отче наш»).
3. **`.env.prod` нужен `docker compose up -d --force-recreate web`.**
   HUP не перечитывает env_file. Изменения только `.py` — HUP норм.
4. **Снапшот с другого компа != прод-конфиг.** Локальные дев-настройки
   часто содержат рестриктивные CORS, выключенный DEBUG, другие SMTP.
   После rsync — диффай `main/settings.py` целиком.
5. **При жалобе «не работает у гостей»** — первым делом CORS preflight,
   не Django логи (CORS блочит на уровне браузера, в Django ничего
   нет).
6. **`grep -c CORS_ALLOWED_ORIGINS settings.py` должно быть `1`.**
   Если 2 — где-то override и беда.
7. **Mobile/manager endpoint полит каждые ~4с** — это основной канал
   обновления чата. `/messages/` мобайл не зовёт. Любой chat-state
   должен быть в `/manager/` ответе.
8. **CheckUp daphne на `127.0.0.1:8000` (host loopback)** — из docker
   bridge напрямую недоступен. Маршрут: через nginx
   (`https://checkupapp.ru/...`) с `LOYALUP_RELAY_REQUIRE_LOOPBACK=False`.
